# 自学习进化方案

## 一、问题定义

当前 skill 的平台操作地图（选择器、输入技巧、已知坑）完全依赖人工手动维护。每次猎聘发版改了 CSS class，或者前程无忧页面结构变了，都要人工发现 → 手动修改 platforms.md。

**核心痛点：**
- 选择器失效只能靠下次使用时「撞上」才发现
- 每个平台的坑全靠人脑记，换对话就丢
- 新平台探索结果没有自动沉淀机制

**目标：** 每次寻访结束后，自动提取本次操作中的有效经验和失败教训，按平台分别写入独立的记忆文件，形成持续进化的知识库。

---

## 二、可行性判断

**结论：技术可行，且实现成本低。**

理由：
1. **数据源已经存在** — 每次寻访的 agent-browser 命令和返回结果就是最原始的学习数据
2. **学习不依赖传统 ML** — 这里的「学习」本质是结构化信息提取 + 规则沉淀，用 LLM 本身就能完成
3. **存储结构天然对齐** — platforms.md 已经是按平台分文件的结构，记忆文件只需沿用
4. **WorkBuddy 已有记忆基础设施** — `.workbuddy/memory/` 目录 + replace_in_file 机制，不需要额外搭建

## 四、核心模块设计

### 模块 1：数据采集

**采集什么：**

| 数据类型 | 采集时机 | 示例 |
|---------|---------|------|
| 选择器命中率 | Phase 2/3 每次提取时 | `.resumeCardWrap--FcnzW` 命中 20/20 vs 0/20 |
| 操作失败记录 | agent-browser 返回异常时 | `fill` 对 React 组件无效、弹窗关闭按钮找不到 |
| 搜索词效果 | Phase 2 每轮搜索后 | 「VLA 具身智能」命中 40 人 vs 「OpenVLA」命中 3 人 |
| 页面结构变化 | 选择器命中率骤降时 | 卡片容器从 `.resumeCardWrap--FcnzW` 变为 `.resumeCardWrap--Xk9mP` |
| 新发现的坑 | 执行过程中任何意外 | 直接 URL 打开详情页触发付费墙 |

**采集方式：**

不需要额外代码。AI Agent 在执行寻访流程时，天然拥有所有交互数据。关键是在 SKILL.md 中**强制要求**在 Phase 4 之后执行 Phase 5 的学习提炼。

### 模块 2：经验提炼

**提炼规则（由 LLM 在 Phase 5 执行）：**

1. **选择器变更检测**
   - 对比本次实际使用的选择器与 platforms.md 中记录的选择器
   - 如果一致 → 无需更新
   - 如果不一致 → 记录新选择器，标记旧选择器为「已失效」
   - 如果完全找不到 → 触发 snapshot 探索，记录新发现

2. **操作模式沉淀**
   - 提炼本次成功/失败的操作模式（如「React 受控组件必须用 nativeInputValueSetter」）
   - 与已有记忆对比，仅增量写入新发现

3. **搜索词效果评估**
   - 记录每轮搜索词 → 结果数 → 匹配度的映射
   - 对同一岗位类型的历史搜索词效果做对比，标记高效/低效词

4. **失败模式归类**
   - 按类型归类：选择器失效、权限限制（付费墙）、反爬触发、页面加载超时
   - 记录触发条件和绕过方案

**提炼输出格式（统一模板）：**

```markdown
## [日期] 学习记录

### 选择器更新
- ❌ 旧：`.resumeCardWrap--FcnzW`（命中率 0/20）
- ✅ 新：`.resumeCardWrap--Xk9mP`（命中率 20/20）

### 新发现
- 直接 URL 打开详情页会触发付费墙，只能通过搜索结果页点击卡片弹窗查看

### 搜索词效果
| 搜索词 | 结果数 | 强推/可联系 | 匹配度 |
|--------|--------|------------|--------|
| VLA 具身智能 机器人 | 40 | 4 | 高 |
| OpenVLA Diffusion Policy | 3 | 0 | 低 |

### 失败记录
- 弹窗关闭按钮 `.ant-lpt-modal-close` 点击无响应，改用 Escape 键成功
```

### 模块 3：记忆写入

**存储结构：**

```
references/memory/
├── liepin.md      # 猎聘网 — 按时间线追加的学习记录
├── 51job.md       # 前程无忧
└── zhipin.md      # Boss 直聘
```

**每个文件的结构：**

```markdown
# 猎聘网 — 自学习记忆

_最后更新：2026-04-10_

## 当前有效选择器

| 用途 | 选择器 | 验证日期 | 验证次数 |
|------|--------|---------|---------|
| 搜索框 | `.searchInput--KgDn1` | 2026-04-10 | 3 |
| 卡片容器 | `.resumeCardWrap--FcnzW` | 2026-04-10 | 3 |
| 详情弹窗 | `.resume-detail-modal-wrap` | 2026-04-10 | 1 |

## 已知坑（最新优先）

1. [2026-04-10] 直接 URL 打开详情页触发付费墙
2. [2026-04-10] 关闭详情弹窗后分页器消失
3. [2026-04-10] React 受控组件必须用 nativeInputValueSetter

## 高效搜索词（按岗位类型）

### 具身智能/VLA 方向
| 搜索词 | 平均结果数 | 平均匹配度 | 使用次数 |
|--------|----------|----------|---------|
| VLA 具身智能 机器人 | 40 | 高 | 2 |
| OpenVLA Diffusion Policy ACT | 11 | 中 | 1 |
| 具身智能 算法研究员 | 20 | 高 | 1 |

## 学习历史（时间线）

### 2026-04-10
- 首次验证详情页弹窗模式，确认选择器有效性
- 发现直接 URL 访问触发付费墙
- 发现关闭弹窗后分页器消失
```

**写入规则：**

1. **追加而非覆盖** — 学习历史用时间线追加，不删旧记录
2. **当前有效选择器可覆盖** — 新选择器验证通过后替换旧的
3. **已知坑去重** — 同一条坑不重复写入，但更新验证日期
4. **搜索词效果累加** — 同一搜索词多次使用时累加统计
5. **验证次数递增** — 每次成功使用后 +1，选择器失效时重置为 0

## 五、数据源文件

| 文件 | 路径 | 用途 | 格式规范 |
|------|------|------|---------|
| 平台记忆文件 | `references/memory/<platform>.md` | 选择器命中率、已知坑、搜索词效果 | 见 `references/memory/liepin.md` |
| 执行日志 | `references/operation_logs.jsonl` | 每次 session 完整轨迹，月度审计数据源 | 见 `references/operation-logs-format.md` |
| 岗位库 | `references/positions.jsonl` | 按岗位记录历史搜索词效果，下次推荐参考 | 见 `references/positions-format.md` |



### 增量更新（每次寻访后自动）

**触发时机：** Phase 4 输出报告后，自动进入 Phase 5

**流程：**
1. 回顾本次所有 agent-browser 交互记录
2. 按提炼规则提取有效经验
3. 读取对应平台的记忆文件
4. 对比去重，仅写入增量信息
5. 更新「当前有效选择器」和统计数字

### 全量刷新（选择器大面积失效时）

**触发条件：**
- 某个选择器连续 2 次命中率 < 50%
- 页面结构明显变化（如 snapshot 输出与记忆文件差异 > 30%）

**流程：**
1. 用 `agent-browser snapshot` 重新探索页面结构
2. 用 `agent-browser eval` 逐个验证关键选择器
3. 重写「当前有效选择器」部分
4. 旧选择器移入「已失效选择器」历史区

### 记忆精简（定期维护）

**触发条件：** 记忆文件超过 200 行

**流程：**
1. 将超过 30 天的学习历史压缩为摘要
2. 仅保留：当前有效选择器 + 已知坑 + 高效搜索词 + 近 30 天详细记录
3. 历史摘要写入文件末尾的「历史摘要」区

## 七、潜在挑战与解决方案

| 挑战 | 影响 | 解决方案 |
|------|------|---------|
| CSS class hash 后缀变化 | 选择器全部失效 | 维护「语义映射表」：用途 → 选择器，而非硬编码。选择器失效时自动触发 snapshot 重建 |
| 学习数据噪声大 | 低效搜索词被错误标记为高效 | 引入「最低使用次数」阈值，搜索词效果需 3 次以上验证才写入高效表 |
| 记忆文件膨胀 | 加载变慢，token 浪费 | 200 行触发精简，30 天以上历史压缩为摘要 |
| 多平台差异大 | 提炼逻辑难以统一 | 每个平台独立记忆文件 + 独立选择器映射，不跨平台泛化 |
| Agent 上下文窗口限制 | 大量历史记忆装不下 | Phase 2/3 执行时只加载「当前有效选择器」部分，学习历史按需查阅 |
| 付费墙/权限变化 | 详情页访问方式突然不可用 | 记录多种访问路径（弹窗/URL/API），按优先级尝试，失败时自动切换 |
