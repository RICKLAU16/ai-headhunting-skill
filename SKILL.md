# AI 猎头寻访 Skill

## 简介

AI 猎头寻访是一个自动化人才搜索与筛选工具。它替代了招聘人员在平台上手动翻页、逐份看简历的重复劳动，将「搜人 → 筛人 → 读简历 → 出报告」的完整寻访流程自动化。

**核心能力：**

- **JD 解析与关键词分流** — 从岗位需求中提炼中英文关键词，自动拆分为 2-3 轮搜索策略（行业术语 + 学术术语 + 岗位名称），最大化覆盖目标人群
- **多轮搜索 + 列表页初筛** — 在招聘平台执行搜索，批量提取候选人信息，按匹配度标记为强推/可联系/待定/不合适
- **详情页深读** — 对强推人选逐个进入简历详情页，提取完整项目经历、技术栈、论文专利、量化业绩，输出真正有判断价值的分析
- **结构化寻访报告** — 输出包含简历编号（resumeId）和详情页链接的完整报告，拿到即可在平台上搜索定位、下载简历、发起沟通

**适用场景：** 硬科技/前沿技术岗位的精准寻访，尤其是候选人稀缺、需要跨领域关键词组合搜索的岗位（如 VLA、具身智能、大模型算法等）。传统关键词单一搜索容易漏人，多轮分流搜索能显著提升覆盖面。

**支持平台：** 猎聘网（已验证），前程无忧（待补充操作地图），Boss 直聘（待补充）。

## 寻访流程

### ⛔ 启动前强制门控

**在开始任何浏览器操作之前，必须完成以下确认步骤，否则不得执行：**

1. 解析用户提供的 JD，提炼：岗位名称、核心技能、城市偏好、薪资范围、经验要求、学历要求
2. 将解析结果整理成结构化摘要，**主动输出给用户**
3. **强制询问用户希望使用的招聘网站**（如猎聘、前程无忧、Boss 直聘等）。若用户未明确指定，**默认使用猎聘网**
4. **等待用户明确回复确认**（如「确认」「OK」「没问题」「开始」等）
5. 用户确认后，方可启动浏览器进入搜索流程

**禁止行为：**
- ❌ 解析完 JD 后直接开始搜索，不等用户确认
- ❌ 用「根据上次确认的参数直接开始」跳过确认步骤（新对话 = 新确认）
- ❌ 假设用户沉默 = 默认同意
- ❌ 使用 headless 模式打开招聘网站

**例外：** 如果用户在同一对话中已确认过需求，且只要求重跑或微调搜索词，则无需重新走完整确认流程，但仍需简要复述关键参数让用户确认。

### ⛔ 浏览器强制规则

**打开任何招聘网站时，必须使用 `--headed` 模式（带界面模式），禁止使用 headless 模式。**

原因：
- 招聘网站需要手动登录（扫码/验证码），headed 模式下用户可直接在浏览器窗口操作
- headless 模式下用户无法完成登录交互，流程会直接卡死
- 正确命令：`agent-browser --headed open <url>`
- ❌ 禁止：`agent-browser open <url>`（默认 headless）

### Phase 1：确认需求

必须输出结构化确认界面（表格格式），禁止纯文字描述。确认完成后方可进入 Phase 2。

**结构化确认界面模板：**

```
┌─────────────────────────────────────────────────────────┐
│ 【JD 分析结果确认】                                      │
├──────────────┬──────────────────────────────────────────┤
│ 岗位名称     │ 数据分析工程师                            │
│ 核心技能     │ Python、SQL、BI 工具（Tableau/PowerBI）   │
│ 城市偏好     │ 深圳、广州（可远程）                       │
│ 薪资范围     │ 20-35K · 13-15薪                         │
│ 经验要求     │ 3-8 年                                   │
│ 学历要求     │ 本科及以上                               │
├──────────────┴──────────────────────────────────────────┤
│ 【候选人画像】                                           │
│ · 典型经历：互联网/金融科技公司数据分析师 / BI工程师       │
│ · 核心能力：Python 数据处理、SQL 复杂查询、BI 可视化     │
│ · 加分项：机器学习基础、A/B 测试经验、数据治理经验        │
├─────────────────────────────────────────────────────────┤
│ 【搜索方案】（选 1 个执行）                              │
│ ☐ 方案 A（精准）：Python AND SQL AND BI                  │
│ ☑ 方案 B（平衡）："数据分析" AND Python    ← 推荐       │
│ ☐ 方案 C（宽泛）：数据分析                              │
├─────────────────────────────────────────────────────────┤
│ 【平台选择】（单选）                                     │
│ ◉ 猎聘（推荐，已验证）                                  │
│ ○ 前程无忧                                              │
│ ○ Boss 直聘                                             │
├─────────────────────────────────────────────────────────┤
│ [确认并开始寻访]    [修改搜索方案]                       │
└─────────────────────────────────────────────────────────┘
```

**关键要求：**
- JD 分析结果用表格输出，清晰对照
- 搜索方案默认推荐「平衡」方案，用户可切换
- **搜索方案必须适配所选平台的搜索语法**（见 `references/platforms.md` 速查表）：
  - 猎聘 → 简化关键词（1-2个词），其余条件靠筛选器
  - 前程无忧 → 可用 AND/OR/NOT 组合，关键词可以更精准
  - Boss 直聘 → 2-3 个核心词，不要堆太多
  - LinkedIn → 字段搜索语法（title/skills/education/company）
- 平台选择默认推荐猎聘，标注平台状态（已验证 / 待补充）
- **必须等用户点击「确认」或回复明确同意词后才进入 Phase 2**

### Phase 2：多轮搜索 + 列表页初筛
- 基于关键词分流策略，设计 2-3 轮搜索词
- 每轮搜索提取列表页数据，**必须包含 `resumeId`（简历编号）和 `resumeUrl`**
- 初筛标记：强推 / 可联系 / 待定 / 不合适

### Phase 3：详情页深读
- 对「强推」和「可联系」人选逐个进入详情页
- 提取完整项目经历、技术栈、论文/专利、量化业绩
- 评估稳定性（在职时长、跳槽频率）

### Phase 4：输出寻访报告

**报告必须包含以下信息：**

#### 强推/可联系人选信息卡（每人一条）
- 姓名（脱敏）
- **简历编号（resumeId）** — ⚠️ 必须输出，方便在平台搜索定位、下载简历、发起联系
- **详情页链接（resumeUrl）** — ⚠️ 必须输出，可直接打开该候选人详情页
- 当前公司 + 职位
- 关键匹配点（为什么推这个人）
- 风险点（跳槽频率、薪资期望、地域等）
- 建议联系优先级（P0/P1/P2）

#### 报告结构
1. 搜索概览（关键词、轮次、结果数）
2. 强推人选详细分析（含简历编号 + 详情链接）
3. 可联系人选简要分析（含简历编号 + 详情链接）
4. 待定人选一览
5. 市场洞察与建议

#### Excel 汇总表（必须同步输出）

每次寻访在生成 Markdown 报告的同时，**必须同步生成一份 Excel 汇总表**，包含所有初筛通过（强推/可联系/待定）的人选数据。

**文件名格式：** `candidates_{platform}_{YYYYMMDD}_{session_id}.xlsx`

**Sheet 1：候选人汇总（所有初筛通过者）**

| 字段 | 说明 |
|------|------|
| 序号 | 排名（按综合评分从高到低） |
| 推荐级别 | P0（强推）/ P1（可联系）/ P2（待定） |
| 姓名 | 脱敏，如「张**」 |
| 简历编号 | resumeId，直接粘贴到平台搜索框可定位 |
| 详情页链接 | resumeUrl，点击直接打开候选人详情 |
| 当前公司 | 最新工作经历公司 |
| 当前职位 | 最新工作经历职位 |
| 工作年限 | 如「5年」 |
| 学历 | 如「硕士·985」 |
| 核心技能 | 逗号分隔 |
| 薪资期望 | 如「30-40K·14薪」 |
| 所在城市 | 如「深圳」 |
| 关键匹配点 | 为什么推/可联系 |
| 风险点 | 跳槽频率/薪资超标/地域等 |
| 综合评分 | 1-10 分，保留一位小数 |

**Sheet 2：搜索词效果（本次执行记录）**

| 搜索轮次 | 搜索词 | 结果数 | 强推数 | 可联系数 | 匹配度评价 |
|---------|--------|--------|--------|---------|-----------|
| R1 | VLA 具身智能 机器人 | 40 | 4 | 2 | 高 |
| R2 | OpenVLA Diffusion Policy | 11 | 0 | 1 | 中 |
| R3 | 具身智能 算法研究员 | 20 | 2 | 1 | 高 |

**生成方式：** 使用 xlsx skill 构建 Excel 文件，写入 `d:\WorkBuddy-data\` 目录。

### Phase 5：自学习进化（每次寻访后自动执行）

**完成后必须执行，不可跳过。** 详细方案见 `references/self-learning.md`。

**1. 更新平台记忆文件（memory/）**
   - 回顾本次所有 agent-browser 交互，提取：
     - 选择器命中率（哪些正常 / 失效）
     - 操作失败记录及替代方案
     - 搜索词效果（每轮 → 结果数 → 匹配度）
     - 新发现的坑或页面行为变化
   - 读取本次平台记忆文件：`references/memory/<platform>.md`
   - 按去重规则写入增量记录

**2. 写入执行日志（operation_logs.jsonl）**

每次寻访后，以一行 JSON 追加写入 `references/operation_logs.jsonl`：

```json
{"session_id":"2026-04-10-001","platform":"liepin","position":"具身智能研究员","jd_summary":"VLA/Diffusion/具身智能算法，3-10年，硕士","keywords":{"R1":"VLA 具身智能 机器人","R2":"OpenVLA Diffusion Policy ACT","R3":"具身智能 算法研究员"},"chosen_plan":"方案 B","timestamp_start":"2026-04-10T22:15:00","timestamp_end":"2026-04-10T22:48:00","duration_min":33,"search_result_count":71,"initial_pass_count":12,"deep_read_count":7,"strong_recommend":4,"contactable":3,"pending":5,"selector_failures":[],"new_discoveries":["直接 URL 打开详情页触发付费墙"],"report_file":"具身智能研究员-寻访报告-v3.md","excel_file":"candidates_liepin_20260410_001.xlsx","user_confirmed":true}
```

**必须包含的字段：**
- `session_id` — 日期 + 当日序号，如 `2026-04-10-001`
- `platform` — 执行平台
- `position` — 岗位名称
- `keywords` — 各轮搜索词
- `duration_min` — 执行耗时（分钟）
- `search_result_count` — 总搜索结果数
- `initial_pass_count` — 初筛通过数
- `deep_read_count` — 深读数
- `strong_recommend / contactable / pending` — 三档推荐数量
- `selector_failures` — 选择器失效记录（空数组表示无）
- `new_discoveries` — 本次新发现的坑
- `report_file` / `excel_file` — 输出文件路径

**3. 更新岗位库（positions.jsonl）**

以岗位名称为主键，追加/更新 `references/positions.jsonl`：

```json
{"position":"具身智能研究员","platform":"liepin","keywords_used":{"R1_VLA具身智能机器人":{"keyword":"VLA 具身智能 机器人","times":2,"avg_result_count":40,"avg_match_rate":"高","last_used":"2026-04-10"},"R2_OpenVLA":{"keyword":"OpenVLA Diffusion Policy ACT","times":1,"avg_result_count":11,"avg_match_rate":"中","last_used":"2026-04-10"},"R3_具身算法":{"keyword":"具身智能 算法研究员","times":1,"avg_result_count":20,"avg_match_rate":"高","last_used":"2026-04-10"}},"candidate_profile":{"experience_min":3,"experience_max":10,"education":"硕士及以上","key_skills":["VLA","Diffusion Policy","模仿学习","ACT"]},"usage_count":2,"last_used":"2026-04-10"}
```

**写入规则：**
- 已有岗位 → 追加搜索词历史，更新 usage_count
- 新岗位 → 新增一行
- 仅记录数据，不做评分算法（轻量版）
