# AI 猎头寻访 Skill

## 简介

AI 猎头寻访是一个自动化人才搜索与筛选工具。它替代了招聘人员在平台上手动翻页、逐份看简历的重复劳动，将「理解岗位 → 设计策略 → 搜人 → 筛人 → 读简历 → 出报告」的完整寻访流程自动化。

**核心能力：**

- **岗位理解分析** — 不是从 JD 摘关键词，而是深度理解岗位要解决什么问题、需要什么能力、什么人能匹配，再从理解反推搜索策略
- **寻访策略设计** — 搜索词从能力模型和人才画像反推，每个搜索词必须说明「目标人群」和「匹配逻辑」，强制配置平台筛选器
- **多轮搜索 + 逐页深读** — 在招聘平台执行搜索，逐页提取候选人信息并立即深读本页强推人选，避免翻页不可回退的问题
- **量化评分框架** — 列表页粗筛三档 + 详情页精评打分，初筛有量化依据，推荐有评分支撑
- **结构化寻访报告** — 输出包含简历编号（resumeId）和详情页链接的完整报告 + Excel 汇总表

**适用场景：** 硬科技/前沿技术岗位的精准寻访，尤其是候选人稀缺、需要跨领域关键词组合搜索的岗位（如 VLA、具身智能、大模型算法等）。

**支持平台：** 猎聘网（已验证），前程无忧（已验证），Boss 直聘（待补充），LinkedIn（待补充）。

## 寻访流程

### ⛔ 启动前强制门控

**在开始任何浏览器操作之前，必须完成以下步骤，否则不得执行：**

1. 执行 **Step 1.1 岗位理解分析**（见下方）
2. 执行 **Step 1.2 寻访策略设计**（见下方）
3. 输出结构化确认界面，**主动展示给用户**
4. **等待用户明确回复确认**（如「确认」「OK」「没问题」「开始」等）
5. 用户确认后，方可启动浏览器进入搜索流程

**禁止行为：**
- ❌ 解析完 JD 后直接开始搜索，不等用户确认
- ❌ 用「根据上次确认的参数直接开始」跳过确认步骤（新对话 = 新确认）
- ❌ 假设用户沉默 = 默认同意
- ❌ 岗位理解分析的任何字段直接复制粘贴 JD 原文（必须经过思考和转化）

**例外：** 如果用户在同一对话中已确认过需求，且只要求重跑或微调搜索词，则无需重新走完整确认流程，但仍需简要复述关键参数让用户确认。

### ⛔ 浏览器通道优先级

**浏览器通道按以下优先级选择：**

1. **MCP integrated_browser**（首选）
   - 无需 headed/headless 配置
   - 用户已提前登录的情况下可直接使用
   - 限制：无 JS eval 能力（数据提取降级到 Level 2）

2. **agent-browser --headed**（备选）
   - 需要 Display 环境
   - 支持 run-code 执行 JS（数据提取 Level 1）
   - 用户可在浏览器窗口手动登录

3. **agent-browser --headless**（最后手段）
   - 无需 Display，但无法让用户手动登录
   - 仅适用于已登录状态（cookie/session 有效）

**执行规则：**
- 启动时先检测 MCP integrated_browser 是否可用
- 如果可用，优先使用 MCP browser
- 如果不可用，尝试 agent-browser --headed
- 如果 headed 不可用，尝试 headless（仅限已有登录态）
- 如果都不可用，提示用户需要手动操作

## Phase 1：岗位理解驱动的寻访计划

### Step 1.1：岗位理解分析

**触发条件：** 每次新对话的首次寻访，强制执行。

**AI 自检清单 — 输出前必须逐条回答：**

1. 这个岗位是在实验室做研究，还是在产线/现场做落地？
   - 如果 JD 中出现「部署」「集成」「落地」「产线」「生产」等词 → 偏落地
   - 如果 JD 中出现「研究」「论文」「发表」「探索」「前沿」等词 → 偏研究

2. JD 中提到的多个技术方向，是否都同等重要？还是有主次之分？
   - 通常 JD 中先提到的职责更重要
   - 「加分项」中的技能权重低于「岗位要求」中的技能

3. 薪资范围对应什么层级的候选人？会不会吸引到过度资深的人？
   - 30-40万在一线城市通常对应中级工程师（3-6年）
   - 如果搜索结果中出现大量 10 年+ 的总监级人选，说明搜索词太泛

4. JD 中有没有「隐含信号」？（具体版本号、特定公司名、特定场景描述）
   - 具体版本号（如 pi0.5）→ 希望候选人对该技术有实际使用经验
   - 特定场景（如「柔性生产」）→ 工业场景，非实验室
   - 特定公司（如「我司自研」）→ 有内部模型团队，这个岗位偏应用侧

5. 这个岗位的理想候选人，最可能在什么类型的公司工作？
   - 从 JD 的技术栈和场景反推目标公司类型
   - 搜索词应该围绕目标公司类型设计，而不是围绕技术关键词堆砌

**输出格式：**

```
┌─────────────────────────────────────────────────────────┐
│ 【Step 1.1 岗位理解分析】                                │
├─────────────────────────────────────────────────────────┤
│ 岗位定位                                                 │
│ · 类型：□ 研究型  □ 工程型  □ 应用落地型  □ 管理型       │
│ · 核心任务：（一句话描述这个岗位要解决什么问题）            │
│ · 产出物：（这个岗位进去后要交付什么）                     │
│ · 难度重心：（这个岗位最难的部分是什么）                   │
├─────────────────────────────────────────────────────────┤
│ 能力模型（加权维度）                                     │
│ · 维度A：[名称]（权重X%）— [需要什么样的能力/经验]        │
│ · 维度B：[名称]（权重X%）— [需要什么样的能力/经验]        │
│ · 维度C：[名称]（权重X%）— [需要什么样的能力/经验]        │
├─────────────────────────────────────────────────────────┤
│ 理想人才画像                                             │
│ · 典型背景：（什么样的公司/团队经历最匹配）               │
│ · 核心证据：（简历上看到什么能证明匹配）                  │
│ · 红旗信号：（什么样的经历说明不匹配）                    │
├─────────────────────────────────────────────────────────┤
│ 硬性过滤条件（命中任一即排除）                            │
│ · [条件1]                                               │
│ · [条件2]                                               │
│ · [条件3]                                               │
├─────────────────────────────────────────────────────────┤
│ 薪资反推分析                                             │
│ · 薪资范围：[JD 薪资]                                    │
│ · 对应层级：[初级/中级/高级/专家/总监]                   │
│ · 对应经验：[X-Y年]                                      │
│ · 超标阈值：[年薪超过多少直接排除]                       │
├─────────────────────────────────────────────────────────┤
│ 隐含信息解读                                             │
│ · [从 JD 的某个细节推导出的隐含要求]                     │
└─────────────────────────────────────────────────────────┘
```

**执行规则：**

1. **岗位定位必须四选一**，不能跳过。大多数技术岗是「研究型 + 工程型」或「工程型 + 应用落地型」的组合，选占比更大的那个
2. **能力模型至少拆 2 个维度，不超过 4 个**。每个维度必须标注权重（总和 100%），权重反映 JD 中该能力的相对重要性
3. **理想人才画像中的「核心证据」必须是可验证的**（如「有 VLA 模型实际使用经验」而非「懂 VLA」）
4. **硬性过滤条件至少 2 条**，从薪资、年龄、学历、岗位类型、经验年限中选取
5. **薪资反推分析必须做**，用薪资范围反推目标候选人的职级和经验区间，避免推荐超标人选
6. **隐含信息解读是加分项**，从 JD 的细节中推导未明说但重要的要求
7. **如果任何字段是 JD 原文复制粘贴的，必须重做** — 岗位理解不是摘抄，是转化

### Step 1.2：寻访策略设计

**触发条件：** Step 1.1 完成后执行。

**与旧版的区别：**

| 维度 | 旧版 | V2 |
|------|------|-----|
| 搜索词来源 | 从 JD 文本中摘抄关键词 | 从能力模型和人才画像反推 |
| 搜索词说明 | 无 | 每个搜索词必须说明「目标人群」和「匹配逻辑」 |
| 搜索轮次 | 固定 2-3 轮 | 根据平台特性灵活调整 |
| 筛选器配置 | 提及但不强制 | 强制配置（岗位类型、薪资范围、经验区间） |
| 非技术岗过滤 | 无 | 搜索阶段通过筛选器前置排除 |

**输出格式：**

```
┌─────────────────────────────────────────────────────────┐
│ 【Step 1.2 寻访策略设计】                                │
├─────────────────────────────────────────────────────────┤
│ 搜索词方案                                               │
│                                                          │
│ R1: [搜索词]                                             │
│   目标人群：[这段搜索要找什么样的人]                      │
│   匹配逻辑：[为什么这个搜索词能找到目标人群]              │
│   预期信噪比：[高/中/低]                                  │
│                                                          │
│ R2: [搜索词]（可选）                                      │
│   目标人群：...                                           │
│   匹配逻辑：...                                           │
│   预期信噪比：...                                         │
├─────────────────────────────────────────────────────────┤
│ 筛选器配置（强制）                                        │
│ · 岗位类型：[排除非技术岗]                                │
│ · 薪资范围：[JD 薪资范围]                                │
│ · 经验区间：[薪资反推的经验区间]                          │
│ · 学历要求：[JD 学历要求]                                │
│ · 其他：[城市、年龄等]                                    │
├─────────────────────────────────────────────────────────┤
│ 平台适配说明                                             │
│ · [所选平台]：[该平台的搜索语法特点和注意事项]            │
└─────────────────────────────────────────────────────────┘
```

**执行规则：**

1. 每个搜索词必须写「目标人群」和「匹配逻辑」，不能只给词不给理由
2. 筛选器配置是强制的，不能跳过。至少配置「岗位类型」和「薪资范围」
3. 搜索词数量和长度根据平台特性调整：
   - 猎聘：1-2 轮，关键词最多 2-3 个（如 `Jetson 嵌入式`），关键词简化 + 筛选器收窄
   - 前程无忧：1-2 轮，关键词最多 3 个空格分隔（如 `VLA 机器人`），不支持布尔运算
   - Boss 直聘：1-2 轮，关键词最多 2-3 个，充分利用筛选器
   - LinkedIn：可多轮，支持字段搜索 + 布尔运算，关键词可更长
4. 如果 R1 搜索结果已足够覆盖（>50 人且信噪比高），R2 可跳过
5. **搜索词验证规则：**
   - 如果搜索词在平台上返回 0 结果，自动放宽或替换，不要跳过
   - 放宽策略：如果搜索词由 3 个以上关键词组成，优先去掉最窄的 1 个词重新搜索
   - 如果放宽后仍为 0，继续去掉次窄的词，直到至少有 1 个关键词
   - 搜索词效果标注验证时间，超过 7 天的标注为「可能过期」

## Phase 2：逐页搜索 + 初筛 + 深读

### 2.1 搜索输入方案

**搜索输入 — 首次输入：**
- 根据所选平台，使用 `references/platforms.md` 中的输入技巧
- 猎聘：React 受控组件必须用 nativeInputValueSetter（见 platforms.md）
- 前程无忧：标准 textbox，可直接 fill

**搜索输入重置 — 多轮搜索时：**

按优先级使用以下方案：

| 方案 | 操作 | 优点 | 缺点 |
|------|------|------|------|
| **方案 A（首选）** | Ctrl+A 全选 → 输入新词替换 | 不重新加载页面，不触发反爬 | 依赖 React 响应 selectAll |
| 方案 B | JS nativeInputValueSetter | 最精确 | 仅 agent-browser CLI 可用；Windows 需文件传递 |
| **方案 C（最后手段）** | 导航到搜索页 URL | 完全重置 | ⚠️ 可能触发猎聘验证码/登录过期 |

**执行规则：**
- **优先使用方案 A（Ctrl+A 替换）**，不触发页面重载，不触发反爬
- 方案 A 操作步骤：`browser_click(ref_search_input)` → `browser_press_key(ref, "Control+a")` → `browser_type(ref, "新搜索词")` → 点击搜索按钮
- 方案 B 仅在方案 A 无法清空旧文本时使用
- **方案 C（导航重置）仅作为最后手段**，仅在页面状态完全异常时使用
- ⚠️ 猎聘反爬敏感，连续导航 2 次以上可能触发验证码（`safe.liepin.com/page/liepin/captchaPage_ip_PC`）
- 导航重置后**必须检查是否跳转到登录页**（如猎聘 `#login`），如果是，提示用户重新登录

**各平台搜索页 URL（仅方案 C 使用）：**
- 猎聘：`https://lpt.liepin.com/search`
- 前程无忧：⚠️ 不能直接导航 URL，必须从左侧导航「人才搜索」进入

### 2.2 操作频率规范

**所有浏览器操作必须遵守以下频率控制，违反会触发平台反爬/验证码：**

| 操作类型 | 等待时间 | 说明 |
|---------|---------|------|
| eval 操作后 | 2-3 秒 | JS 执行后等待页面响应 |
| 按钮点击后 | 2-3 秒 | 等待页面渲染结果 |
| 页面导航后 | 5 秒 | 等待网络加载 + 反爬规避 |
| 关闭弹窗后 | 2-3 秒 | 等待页面状态恢复 |
| 连续操作满 10 次 | 主动暂停 10 秒 | 给平台缓冲窗口 |

**⚠️ 严禁连续快速操作。猎聘反爬敏感，短时间内多次导航+eval 会触发验证码。**

### 2.3 数据提取方案

根据浏览器通道能力，按优先级选择：

**Level 1（最优）：JS eval 批量提取**
- 适用：agent-browser CLI 的 run-code
- 效率：一次提取整页 20 人所有信息（含 resumeId）
- 提取脚本见 `references/platforms.md` 各平台「完整提取脚本」

**Level 2（降级）：snapshot 文本解析 + get_attribute 补充**
- 适用：MCP browser（无 JS eval）
- 流程：
  1. browser_snapshot 获取整页候选人文本信息
  2. 解析 snapshot YAML 中的候选人信息
  3. 对每位候选人用 browser_get_attribute 获取 resumeId
- 效率：1 次 snapshot + N 次 get_attribute

**Level 3（兜底）：纯 snapshot 解析**
- 适用：get_attribute 也不可用时
- 流程：仅从 snapshot 文本中提取信息，resumeId 留空
- 注意：resumeId 为空时必须在报告中标注「需手动搜索定位」

**执行规则：**
- 优先尝试 Level 1
- 如果 JS eval 不可用，自动降级到 Level 2
- **禁止在无 JS eval 时逐个点击候选人获取列表页信息**（效率极低）
- Level 2 中 snapshot 已包含大部分信息，get_attribute 仅用于补充 resumeId

### 2.4 前置过滤规则

**列表页初筛时，对每位候选人快速判断：**

**直接排除（非技术岗）：** 职位名称包含以下关键词 → 标记「不合适」：
- HR、人力资源、招聘、投资、财务、行政、法务
- 项目管理（非技术方向）、产品经理（非技术方向）

**搜索阶段前置排除：** 通过平台筛选器的「职位类型」排除非技术岗（如果平台支持）。

### 2.5 列表页粗筛三档

**列表页信息不足以支撑精确评分，使用粗筛三档：**

| 推荐级别 | 判定标准 | 后续动作 |
|---------|---------|---------|
| **P0 强推** | 核心技能高度匹配 + 薪资在范围 + 经验对口 | 本页深读详情页 |
| **P1 可联系** | 部分核心技能匹配 + 薪资在范围 | 本页深读详情页 |
| **P2 待定** | 有一定关联但匹配度不高 | 仅记录，不深读 |
| 不合适 | 非技术岗 / 薪资远超预算 / 硬性条件不符 | 不进入报告 |

**粗筛核心判断逻辑（按优先级）：**
1. 岗位类型是否匹配？（非技术岗直接排除）
2. 核心技能是否出现？（对照 Step 1.1 能力模型）
3. 薪资期望是否在预算范围内？
4. 经验年限是否在 Step 1.1 薪资反推的区间内？

### 2.6 逐页深读流程

**核心改造：Phase 3 深读融入 Phase 2 逐页流程，不再独立执行。**

```
逐页执行流程：
  第1页 → 提取列表数据 → 粗筛标记 → 对本页 P0/P1 人选深读详情页 → 翻到第2页
  第2页 → 提取列表数据 → 粗筛标记 → 对本页 P0/P1 人选深读详情页 → 翻到第3页
  ...
  最后一页 → 提取列表数据 → 粗筛标记 → 对本页 P0/P1 人选深读详情页 → 结束
```

**详情页深读操作：**
- 触发方式：点击候选人卡片内容区域（猎聘：`.resumeCardContent--A03AZ`）
- 等待 2-3 秒让弹窗/新页面加载
- 提取详情内容（猎聘：`.resume-detail-modal-wrap` 的 innerText）
- 内容可能较长，分段提取：`.substring(0, 3000)` / `.substring(3000, 6000)`
- 关闭弹窗/新 tab 后，等待 2-3 秒再继续
- **连续操作间隔 2-3 秒**，避免触发反爬

**异常恢复：**
- 关闭弹窗后如果分页器消失 → 用导航重置方案恢复到当前页
- 深读弹窗导致页面状态异常 → 导航重置恢复
- 翻页后无法回退 → 设计上已规避（逐页深读）

### 2.7 详情页精评打分

**对深读后的候选人进行精评打分（仅 P0/P1 人选）：**

```
综合评分 = Σ(维度得分 × 维度权重)

维度定义（默认权重，根据 Step 1.1 能力模型调整）：

┌──────────────┬────────┬──────────────────────────────────────┐
│ 维度          │ 权重   │ 评分标准（1-10分）                    │
├──────────────┼────────┼──────────────────────────────────────┤
│ 技能匹配度    │ 40%    │ 10: 完美匹配所有核心技能              │
│              │        │ 7-9: 匹配大部分核心技能                │
│              │        │ 4-6: 匹配部分核心技能                  │
│              │        │ 1-3: 几乎不匹配                        │
├──────────────┼────────┼──────────────────────────────────────┤
│ 经验匹配度    │ 25%    │ 10: 经验完美对标岗位需求              │
│              │        │ 7-9: 经验高度相关                      │
│              │        │ 4-6: 经验部分相关                      │
│              │        │ 1-3: 经验不相关                        │
├──────────────┼────────┼──────────────────────────────────────┤
│ 薪资匹配度    │ 20%    │ 10: 期望薪资在预算中段                │
│              │        │ 7-9: 期望薪资在预算范围内              │
│              │        │ 4-6: 期望薪资略超预算（<20%）          │
│              │        │ 1-3: 期望薪资远超预算（>20%）          │
├──────────────┼────────┼──────────────────────────────────────┤
│ 学历+背景    │ 15%    │ 10: 学历超要求 + 顶尖公司背景          │
│              │        │ 7-9: 学历达要求 + 知名公司背景          │
│              │        │ 4-6: 学历达要求                        │
│              │        │ 1-3: 学历不达要求                      │
└──────────────┴────────┴──────────────────────────────────────┘
```

**推荐级别精评判定：**

```
综合评分 ≥ 8.0  → P0 强推
综合评分 6.5-7.9 → P1 可联系
综合评分 5.0-6.4 → P2 待定
综合评分 < 5.0  → 不合适（不进入报告）

一票否决（任一命中直接降级或不推荐）：
· 薪资期望超过预算 30% 以上 → 降一级
· 薪资期望超过预算 50% 以上 → 不推荐
· 年龄超过 Step 1.1 设定的硬性过滤线 → 不推荐
· 纯管理岗/非技术岗 → 不推荐
· 工作经验 < 1年（应届/实习）→ 降一级
```

**维度权重可定制：** Step 1.1 的能力模型应映射到评分维度。如果能力模型与默认权重差异较大，应在 Step 1.2 中说明调整后的评分权重。

## Phase 3：输出寻访报告

**报告必须包含以下信息：**

#### 强推/可联系人选信息卡（每人一条）
- 姓名（脱敏）
- **简历编号（resumeId）** — ⚠️ 必须输出，方便在平台搜索定位、下载简历、发起联系
- **详情页链接（resumeUrl）** — ⚠️ 必须输出，可直接打开该候选人详情页
- 当前公司 + 职位
- 关键匹配点（为什么推这个人）
- 风险点（跳槽频率、薪资期望、地域等）
- 综合评分（1-10 分，保留一位小数）
- 建议联系优先级（P0/P1/P2）

#### 报告结构
1. 搜索概览（关键词、轮次、结果数）
2. 强推人选详细分析（含简历编号 + 详情链接 + 综合评分）
3. 可联系人选简要分析（含简历编号 + 详情链接 + 综合评分）
4. 待定人选一览
5. 市场洞察与建议

#### Excel 汇总表（必须同步输出）

**文件名格式：** `candidates_{platform}_{YYYYMMDD}_{session_id}.xlsx`

**输出路径：** `{OUTPUT_DIR}/`

**Sheet 1：候选人汇总（所有初筛通过者）**

| 字段 | 说明 |
|------|------|
| 序号 | 排名（按综合评分从高到低） |

## Phase 4：自学习进化（每次寻访后自动执行）

**完成后必须执行，不可跳过。** 详细方案见 `references/self-learning.md`。

⚠️ **强制检查：Phase 4 必须完成以下三件套，缺一不可：**
1. ✅ 更新平台记忆文件（memory/）
2. ✅ 写入执行日志（operation_logs.jsonl）
3. ✅ 更新岗位库（positions.jsonl）

**如缺少任意一项，视为 Phase 4 未完成，必须在输出报告前补齐。**

**1. 更新平台记忆文件（memory/）**
   - 回顾本次所有浏览器交互，提取：
     - 选择器命中率（哪些正常 / 失效）
     - 操作失败记录及替代方案
     - 搜索词效果（每轮 → 结果数 → 匹配度）
     - 新发现的坑或页面行为变化
   - 读取本次平台记忆文件：`references/memory/<platform>.md`
   - 按去重规则写入增量记录

**2. 写入执行日志（operation_logs.jsonl）**

每次寻访后，以一行 JSON 追加写入 `references/operation_logs.jsonl`：

```json
{"session_id":"2026-04-10-001","platform":"liepin","position":"具身智能研究员","jd_summary":"VLA/Diffusion/具身智能算法，3-10年，硕士","keywords":{"R1":"VLA 具身智能 机器人","R2":"OpenVLA Diffusion Policy ACT","R3":"具身智能 算法研究员"},"chosen_plan":"方案 B","timestamp_start":"2026-04-10T22:15:00","timestamp_end":"2026-04-10T22:48:00","duration_min":33,"search_result_count":71,"initial_pass_count":12,"deep_read_count":7,"strong_recommend":4,"contactable":3,"pending":5,"selector_failures":[],"new_discoveries":["直接 URL 打开详情页触发付费墙"],"report_file":"具身智能研究员-寻访报告-v3.md","excel_file":"candidates_liepin_20260410_001.xlsx","user_confirmed":true}
```

**必须包含的字段：**
- `session_id` — 日期 + 当日序号，如 `2026-04-10-001`
- `platform` — 执行平台
- `position` — 岗位名称
- `keywords` — 各轮搜索词
- `duration_min` — 执行耗时（分钟）
- `search_result_count` — 总搜索结果数
- `initial_pass_count` — 初筛通过数
- `deep_read_count` — 深读数
- `strong_recommend / contactable / pending` — 三档推荐数量
- `selector_failures` — 选择器失效记录（空数组表示无）
- `new_discoveries` — 本次新发现的坑
- `report_file` / `excel_file` — 输出文件路径

**3. 更新岗位库（positions.jsonl）**

以岗位名称为主键，追加/更新 `references/positions.jsonl`：

```json
{"position":"具身智能研究员","platform":"liepin","keywords_used":{"R1_VLA具身智能机器人":{"keyword":"VLA 具身智能 机器人","times":2,"avg_result_count":40,"avg_match_rate":"高","last_used":"2026-04-10"}},"candidate_profile":{"experience_min":3,"experience_max":10,"education":"硕士及以上","key_skills":["VLA","Diffusion Policy","模仿学习","ACT"]},"usage_count":2,"last_used":"2026-04-10"}
```

**写入规则：**
- 已有岗位 → 追加搜索词历史，更新 usage_count
- 新岗位 → 新增一行
- 仅记录数据，不做评分算法（轻量版）
